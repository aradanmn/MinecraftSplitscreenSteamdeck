## What Was Accomplished

### 1. Controller isolation test in minecraftSplitscreen.sh

The test replaces the prior device-listing windows with live button press
reporting. Each half of the screen shows:

- **Header**: which `/dev/input` devices are visible inside that slot's bwrap sandbox
- **Scrolling log**: live button press / release events read from the event device inside bwrap

Same Steam shortcut, no argument changes. Auto-detect logic unchanged:
- gamescope context → `nestedPlasma()` → nested KWin
- KDE context → `runIsolationTest()`

### 2. evdev button reader (`/tmp/event_reader.py`)

Runs inside bwrap. Reads raw 24-byte `input_event` structs (`llHHi`) from the
event device. Filters for `EV_KEY` (type=1) press/release events and prints
them as human-readable lines:

```
[20:45:03]  A           PRESS
[20:45:03]  A           release
```

Button map covers A/B/X/Y, L1/R1/L2/R2, Select/Start/Guide, L3/R3, D-pad.

### 3. GTK window script (`/tmp/splitscreen_window.py`)

Takes CLI args: `slot y_pos width height js_dev ev_dev`. Builds bwrap prefix
internally from the device args. Uses `GLib.io_add_watch` on a non-blocking fd
to stream subprocess output into a `Gtk.TextView` without blocking the GTK
main loop.

### 4. Critical fix: python3 -c quoting

Initial attempt embedded Python code inline via `python3 -c "..."` with bwrap
commands interpolated via shell variables. When the bwrap command contained
single quotes (e.g. `tr '\n' '  '`), it caused a `SyntaxError: unterminated
string literal`.

**Fix**: write both Python scripts to `/tmp` using `cat > /tmp/script.py <<'PYEOF'`
heredocs, then call them with clean argument lists. No quoting interaction
between shell and Python at all.

## Known Limitation Identified (end of session)

D-pad and joystick axes **do not register** in the current test. Root cause:

- Most controllers (including Steam Deck pads) report d-pad as `EV_ABS` hat axes
  (`ABS_HAT0X`=16, `ABS_HAT0Y`=17) with values -1/0/+1, not as `EV_KEY` buttons
- The `BTN_DPAD_*` codes (544-547) only apply to controllers that report d-pad
  digitally — uncommon
- Joystick axes are `EV_ABS` (ABS_X/Y, ABS_RX/RY etc.) and continuous — noisy
- Current `event_reader.py` only handles `EV_KEY` (type=1)

**Next step**: add `EV_ABS` handling — report hat axis direction changes for d-pad,
and threshold-gated changes for analog sticks/triggers.

## Key Technical Details

### bwrap isolation command pattern

```bash
bwrap \
  --dev-bind / /    \   # full host rootfs
  --dev /dev        \   # replace /dev with minimal devtmpfs (no input devices)
  --dev-bind /dev/input/eventN /dev/input/eventN \
  --dev-bind /dev/input/jsN   /dev/input/jsN     \
  -- python3 /tmp/event_reader.py /dev/input/eventN
```

### js → event pair mapping (sysfs)

```bash
for js in /dev/input/js*; do
    jsnum="${js##*js}"
    evname=$(ls /sys/class/input/js${jsnum}/device/ | grep '^event' | head -1)
    echo "$js /dev/input/$evname"
done
```

### input_event struct (64-bit Linux)

```
struct input_event {
    long tv_sec;    // 8 bytes
    long tv_usec;   // 8 bytes
    __u16 type;     // 2 bytes  — EV_KEY=1, EV_ABS=3
    __u16 code;     // 2 bytes  — button/axis code
    __s32 value;    // 4 bytes  — 0=release, 1=press, -1..+1 for axes
};
// Total: 24 bytes, unpack with struct.unpack('llHHi', data)
```
