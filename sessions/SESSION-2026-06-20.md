## RAW TRANSCRIPT PUSH 3 — FIFO consumer fix (DISPLAY_MODE_CHANGE was killing docked_flow)

User: "I launched a new test Minecraft hasn't quit yet"

→ Found the actual bug: Test 1 was injecting DISPLAY_MODE_CHANGE handheld at the start. docked_flow's _handle_msg for handheld mode returns 1, causing docked_flow's event loop to `return 1` (line 373), which exits the background docked_flow process entirely. After that, no FIFO consumer exists, so all subsequent _inject calls (CONTROLLER_ADD, SLOT_DIED) block forever because there's no reader.

→ The test log confirms: "Slot 1 active after 0s" (from state file check during CONTROLLER_ADD's spawn_instance), "Minecraft ready after 25s" (real instance loaded). Then SLOT_DIED wrote to the FIFO but nobody read it for 200s → timeout.

→ Fixed: removed DISPLAY_MODE_CHANGE injects from all tests except Test 5 (which specifically tests the docked→handheld transition). The orchestrator starts in docked mode (set by launchTestFromPlasma's state file init). Tests that use a single slot (Test 1) work fine in docked mode — it's just one controller.

→ Real load timing confirmed: Minecraft instance loaded in 25s under real APU render contention. The hardcoded 180s timeout is safe (6x margin).

Tests that still use DISPLAY_MODE_CHANGE:
- Test 5: docked → handheld (legitimate mode transition test)
- All others: removed
