## The question that framed the session

User: "where are we at here — I was under the assumption the test code calls the same APIs
production would, so it'd be an easy switch, but the bugs we fixed are coming back."

Answer (grounded by reading the code, not theory): the test and production paths **share the
consumer half** (orchestrator → spawn_instance → window_manager → KWin — the windowing engine,
which is why test 4 validated on screen) but the test harness **fakes the producer half**.
`tests/test_phase_b_lifecycle.sh` injects hand-written **2-field** `CONTROLLER_ADD /dev/null
/dev/zero` straight into the FIFO, bypassing `controller_monitor` entirely. The real monitor
emits **4 fields** (`event js vendor product`). So the orchestrator's `${msg_arg#* }` parse —
fine on 2 fields — polluted `js_node` with the trailing vendor/product on the real message.
That's bug C1, and it was *structurally invisible* to the test. The "bugs coming back" were
never the windowing bugs (those stayed fixed); they were a different class living only in the
bypassed producer code.

### NEW production bug found live — CONTROLLER_REMOVE (a C1 twin on the remove path)
Disconnecting controllers did nothing — instances stayed up, survivors never re-tiled.
Root cause: `controller_monitor` emits `CONTROLLER_REMOVE <event_node>` (a /dev/input
path), but the orchestrator handler treated the arg as a SLOT NUMBER, so
`slot_is_active "/dev/input/event26"` was false → "ignoring". The test harness injects
`CONTROLLER_REMOVE <slot>` (a number), so it never caught it — the SAME producer/consumer
wire-format seam that hid C1, and the 8 module-auditors missed it because they checked
each module's internal consistency, not the cross-module wire format the test faked.
FIX (6c24642): handler is now format-aware — bare slot number OR device node (resolved
via new `_find_slot_by_event_node`). Deployed + verified on the Deck. (In-game quit was
unaffected — that path goes watchdog → `SLOT_DIED <slot>`, already a slot number.)
