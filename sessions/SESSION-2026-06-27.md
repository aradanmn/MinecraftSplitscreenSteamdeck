## Windowing: force re-composite on placement — black-half occlusion fix (06:06)

Docked regression test surfaced a pre-existing windowing bug (NOT from the controller work):
when a new player's window spawns at 0,0 over an existing one, KWin stops painting the covered
window (occlusion culling) → it goes BLACK and never repaints; the re-tile moves it but
kwin_place_windows sets frameGeometry only (no raise/repaint), so the black half persists.
Only the FIRST-covered window (P1) was affected; P2/P3/P4 fine. Fix: raise each window after
setting frameGeometry in the KWin script (kwin_positioner.sh) so KWin re-composites it.
NOT yet validated on hardware. If raise alone is insufficient, escalate to a remap kick.

## Windowing/gamescope deep-research → docs/RESEARCH-WINDOWING-GAMESCOPE-2026-06-27.md (06:39)

Cited research on the 3 windowing problems. Key: (1) black-out = Wayland frame-callback
starvation of fully-occluded surfaces — MOVE doesn't repaint (matches us), RESIZE-jiggle or
minimize-toggle does, raise is racy (explains 3-not-4); occlusion can't be globally disabled
on KWin6 Wayland. (2) center-spawn = a FORCED kwinrulesrc rule (placement=5 Centered,
placementrule=2 Force) is the only true map-time fix — VERIFY WM_CLASS via xprop first.
(3) gamescope reset = memory pressure (JVM heap 1536×4=6GB, NOT swapchain) + WSI-layer abort
(Vulkan-only, from nested compositor/zink not the GL game) → DISABLE_GAMESCOPE_WSI=1 (not
ENABLE=0) + lower -Xmx + RADV_SYS_MEM_LIMIT=50. 7-step ordered plan in the doc. Next: draft
the center-spawn rule as isolated new code; other fixes = apply+test together on return.
