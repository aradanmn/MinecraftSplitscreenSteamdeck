
## Handheld: launch WITHOUT bwrap (full access) — maintainer's idea (2026-06-27 ~05:40)

Porous-sandbox fix was wrong: in handheld slot 1 binds NO js, and --dev /dev empties
/dev/input, so the built-in's Steam 28de:11ff virtual has no device node to open → SDL3
(Controlify uses SDL 3.2.18) reports "No controllers found." Maintainer's call: handheld is
one player, nothing to isolate → don't sandbox it at all. New _build_direct_command (no
bwrap, env + launcher, full system access, ALLOW_STEAM_VIRTUAL_GAMEPAD=1); spawn_instance
runs it when js_node is empty (handheld). Docked (js bound) keeps the bwrap sandbox
untouched. Plus prior commit: handheld doesn't start the hotplug monitor (no duplicate slot).
NOT yet re-validated undocked.

**VALIDATED 05:45** — handheld direct launch confirmed on Deck: mode=handheld, single active
slot (no duplicate), java=1/bwrap=0 (no sandbox), built-in controls the one window, Controlify
shows correct Steam Deck icons. Both controller modes now solid: docked (4-DS4 raw bind) +
handheld (direct). Follow-up (harmless): the porous-gating in _build_bwrap_command is now dead
(handheld uses _build_direct_command, so _build_bwrap_command always gets a bound js) — remove
in the deferred cleanup.

## Windowing: force re-composite on placement — black-half occlusion fix (06:06)

Docked regression test surfaced a pre-existing windowing bug (NOT from the controller work):
when a new player's window spawns at 0,0 over an existing one, KWin stops painting the covered
window (occlusion culling) → it goes BLACK and never repaints; the re-tile moves it but
kwin_place_windows sets frameGeometry only (no raise/repaint), so the black half persists.
Only the FIRST-covered window (P1) was affected; P2/P3/P4 fine. Fix: raise each window after
setting frameGeometry in the KWin script (kwin_positioner.sh) so KWin re-composites it.
NOT yet validated on hardware. If raise alone is insufficient, escalate to a remap kick.
