I had insisted a real splitscreen launch was needed to confirm what controllers the
orchestrator sees. Maintainer corrected: **the orchestrator enumerates `/proc/bus/input/
devices` at the HOST level, before any bwrap** — bwrap only filters what each instance sees
afterward. So enumeration is a pure function of the host device tree: readable now, and
unit-testable via the module's `PROC_INPUT_DEVICES` override. No launch required.

## Display Incident Log (Evening session — stock SteamOS, no custom code)

### Context

Factory-reset Steam Deck OLED. Observing external display (Sceptre O34 ultrawide via
CalDigit dock, USB-C DP alt-mode) failure behavior on stock SteamOS. Building reproducible
incident log for Valve issue filing.

### Root Cause (Confirmed)

Every blank-display incident is accompanied by:
```
[drm] Failed to add display topology, DTM TA is not initialized.
```

The AMD PSP's DTM TA (Display Topology Management Trusted Application) isn't initialized
when the kernel's MST probe fires on boot or resume. Gamescope selects DP-1 and sets the
mode but signal never reaches the display.

### Recovery Procedure (Confirmed Working)

```bash
systemctl --user restart gamescope-session.target
```

- Works every time across all 9 incidents
- Exit code 1 ("dependency job failed") is normal, does NOT prevent recovery
- Haptic motor whirr = session reinitializing = display coming back
- Dock replug NOT required

### Incident Tally

| # | Trigger | Dock replug tried | Restarts needed | Exit codes | Result |
|---|---------|-------------------|-----------------|------------|--------|
| 1 | Cold boot with dock connected | Yes | 1 | 1 | ✅ |
| 2 | Post-sleep wake | Yes | 1 | 1 | ✅ |
| 3 | Post-sleep wake | Yes | 1 | 1 | ✅ |
| 4 | Post-sleep, dock plugged in after wake | No | 1 | 1 | ✅ |
| 5 | Post-sleep, dock plugged in after wake | No | 1 | 1 | ✅ |
| 6 | Cold boot with dock connected | No | 1 | 0 | ✅ |
| 7 | Post-sleep, dock disconnected + reconnected | No | 2 | 1, 1 | ✅ |
| 8 | Cold boot with dock connected | No | 1 | 1 | ✅ |
| 9 | Post-sleep wake (dock connected through sleep) | Yes (HDMI cable ×3) | 1 | 1 | ✅ |

**Pattern:** dock replug does NOT fix the issue. `systemctl --user restart gamescope-session.target` is the consistent fix.

#### Incident #9 detail (richest capture so far — 2026-06-25 ~22:32)

Resume from sleep (`PM: suspend exit` 22:27:31) with dock connected through sleep. External
display blank; maintainer replugged the HDMI cable 3× with no effect.

Live evidence captured before recovery (deck@192.168.1.131, SteamOS 3.8.11):
- **Kernel sees the monitor:** `/sys/class/drm/card0-DP-1/status: connected`.
- **gamescope selects it, then DTM TA fails:** the journal shows
  `drm: Connector DP-1 -> SPT - Sceptre O34` then `drm: selecting connector DP-1`
  (22:32:21), each immediately followed by
  `[drm] Failed to add display topology, DTM TA is not initialized.`
- **The 3 replugs are in the log** as disconnect/reconnect cycles 22:31:25 → 22:32:08
  (`current connector 'DP-1' disconnected` → falls back to `selecting connector eDP-1`)
  → 22:32:21 (reconnect → `selecting connector DP-1`), every cycle ending in the DTM TA
  error. So replug re-probes the connector but never initializes the DTM TA.
- **kscreen-doctor only enumerated the internal output** (`gamescope` 1280x720) while blank
  — the selected DP-1 never produced a usable topology.

This is the key takeaway for #2363: the failure is **not** EDID/hotplug detection (connector
reads `connected`, gamescope picks the correct Sceptre) — it is purely the PSP DTM TA being
uninitialized at resume, which `systemctl --user restart gamescope-session.target` fixes by
reinitializing the session/topology path. Recovery exit code 1, display confirmed back by
maintainer.

### Lifecycle flaw — maintainer insight → new v1-blocker (#37)

would kill a player mid-session — fatal for a LAN host (kills the world for all). Fix:
teardown on game-window-destroy (then SIGTERM→grace→SIGKILL, making Bug B non-fatal); make
controller-disconnect tolerant (keep instance, allow reconnect). Defuses #14 and #15.

## RAW TRANSCRIPT PUSH 3 — #37 lifecycle decoupling implemented (commit 6adab75)

Started #37 (the keystone fix surfaced by the live session). Read the lifecycle code first
(watchdog pid-check, orchestrator CONTROLLER_REMOVE/SLOT_DIED handlers, dex window primitives),
confirmed the design with the maintainer (incl. two accepted v1 limitations: live-rebind of a
reconnected pad is out of scope — bwrap mounts fixed at launch; walk-away idle instance lingers).

Implemented:
- **watchdog.sh** — for each active slot with a recorded `wid`, check whether that window is
  still in the live tree via `dex_list_windows` (a full recursive XQueryTree walk — sees
  override-redirect windows). Confirmed-absent for `WATCHDOG_WINDOW_GONE_TICKS` consecutive
  polls → treat as death → `SLOT_DIED` → `teardown_instance` force-kills (so a Bug-B hung JVM
  can't wedge the session). Helper `_watchdog_window_present` returns 0/1/2; **2 = can't tell**
  (no wid yet / dex unavailable / empty listing) → never escalates, so it can't kill a
  launching or unverifiable instance.
- **orchestrator.sh** — `CONTROLLER_REMOVE` now PRESERVES the instance (logs, no teardown);
  player-leave flows through window-gone instead. Protects a LAN host's world from a dead
  battery.
