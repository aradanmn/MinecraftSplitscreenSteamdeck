## RAW TRANSCRIPT PUSH 7 — 2026-06-21 (continued session)

### Problem: Orchestrator blocked, T1.3/T1.4 failing

The core bug causing T1.3 (slot inactive within 30s of kill) and T1.4 (PID reaped)
to always fail was identified and fixed this session.

ROOT CAUSE: `spawn_instance 2>&1 | sed 's/^/[orchestrator] /' >&2`
The sed pipe keeps its write-end open as long as ANY process inherits it — including
bwrap, PolyMC, and Java (descendants of spawn_instance's setsid). So the orchestrator
was blocked in that pipe for the entire lifetime of Minecraft. It couldn't read
`SLOT_DIED` from the FIFO until AFTER Minecraft was already dead — defeating T1.3's
30-second window.

Prior fix (375693b): Redirected bwrap's stdout/stderr to $LOG. This helped but the
orchestrator was STILL in the pipe because spawn_instance itself (doing _poll_for_java,
_poll_for_window, sync_apply_layout) continued running inside the pipe for ~25-35 more
seconds. Those polling steps ran inside spawn_instance's shell which held the pipe
write-end — regardless of where bwrap output was going.

THE FIX (1b06f5c — modules/orchestrator.sh):
Replace the sed-pipe pattern with a temp-file pattern:
  ```
  _si_log=$(mktemp /tmp/spawn_instance_slot${slot}_XXXXXX.log)
  spawn_instance "$slot" ... >"$_si_log" 2>&1 || true
  sed 's/^/[orchestrator] /' < "$_si_log" >&2
  rm -f "$_si_log"
  ```
Now: spawn_instance runs with NO pipe. When it returns, the orchestrator immediately
proceeds to `sleep 3; _reflow_layout` and then enters `_read_fifo_msg`. The watchdog
has already written SLOT_DIED by then (within 2s of bwrap dying). Orchestrator reads
it, calls teardown_instance, sets active=false. T1.3 should pass.

### User feedback
User noted sessions were spinning too long on the pipe issue. Redirected toward
the correct architectural fix (temp file, no pipe) rather than continuing to
patch the bwrap side.

- 1b06f5c  fix(orchestrator): use temp file for spawn_instance output, not a pipe

### Blocker resolution: testDirect early exit
SSH background (`&`) and `nohup+disown` both killed the bash process during
dex.sh heredoc write. Tmux sessions died after `sleep 2` from an unknown signal.

ROOT CAUSE (for SSH methods): despite `huponexit=off`, something was sending
SIGKILL (cat wrote 0 bytes before dying). Nohup+disown didn't protect against it.

ROOT CAUSE (for tmux): unknown signal kills `sleep 2` in the testDirect case
~0.3s after starting. The EXIT trap fired without any `SCRIPT_DIR=` trace line
appearing, confirming sleep was interrupted before the next command.

SOLUTION: `systemd-run --user --scope --collect -E DISPLAY=:0 -E XAUTHORITY=...`
Creates a transient scope unit that's truly independent of the SSH session.
Confirmed working: 112KB debug log created, multiple bash processes alive.

### Root cause of cleanup failures
`teardown_instance` sends SIGTERM then waits up to `INSTANCE_LIFECYCLE_TEARDOWN_GRACE_S=10`
seconds for the process to die, then SIGKILL. Total: ~11s minimum.

Per-test cleanups used `_wait_for_slot_inactive 1 15` (15s) — borderline.
`_clean_instances` only `sleep 5` — not enough for full teardown.

If cleanup doesn't finish in time, the next test gets a `_find_free_slot` that
skips slot 1 (still active) and spawns to slot 2 instead. `_wait_for_slot_active 1 30`
then times out → cascade failure.

### Fix (commit 3679156)

2. All per-test cleanup waits: changed from 15s → 30s, all with `|| true` so

## RAW TRANSCRIPT PUSH 10 — FIFO open-block hang + testDirect process leak

### Live incident reported by user
User observed on the Deck screen: Minecraft in bottom half, a black window
covering the right half, desktop visible in the upper-left quadrant.

### Diagnosis
Not a working splitscreen — the wreckage of the 18:25 `testDirect all` run that
HUNG and never exited:
- The systemd scope `run-p219461` was still "active (running)" 59 min later.
- 57 stray `minecraftSplitscreen.sh testDirect all` bash procs (all ~48 min old,
  blocked).
- 2 orphan `SplitscreenBlack` tkinter placeholders (slots 2 & 4 of a prior
  4-player grid) reparented to init — the black right half.
- Slot 1's Minecraft window was IsUnMapped — desktop showed through top-left.
- Slot 2's window correctly mapped to the bottom half.

Cleaned up live: stopped the scope, killed the 57 procs + placeholders + stray
Minecraft, reset the stale state file (slots 1&2 still flagged active), removed
the stale FIFO.

### Root cause
- `_read_fifo_msg` opened the FIFO read-only (`< "$fifo"`). A read-only FIFO open
  blocks in open() until a writer connects, and that block precedes `read -t`, so
  with no writer the timeout never engaged → orchestrator iteration wedged forever.
- testDirect cleanup used `pkill -P` (direct children only) → docked_flow
  grandchildren blocked on the FIFO orphaned to init and accumulated.

### Fixes
- `modules/orchestrator.sh`: `_read_fifo_msg` opens read-write (`<>`) so open()
  returns immediately and `-t` governs the wait.
- `minecraftSplitscreen.sh` testDirect: recursive `_kill_tree` (TERM→KILL) +
  remove FIFO last in `_td_cleanup`, used by both the EXIT trap and post-test.
- `tests/test_phase_b_lifecycle.sh`: removed T5 from `all` suite (needs physical
  undock); every test fn returns 0 on its cleanup path; early-bail guards return 0
  (no more double-counted "exited with error"); `_wait_for_slot_inactive` logs
  its timeout.
