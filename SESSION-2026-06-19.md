## The Handoff's Claims (what we were told)

1. **Bug #1 — GPU stripped:** `--dev /dev` strips `/dev/dri/*`; Qt xcb can't init →
   PolyMC exits silently. Suggested fix: add `--dev-bind /dev/dri /dev/dri`.
2. **Bug #2 — SingleApplication:** PolyMC's `QLocalServer` uses abstract Unix sockets
   in the network namespace, so slots 2–4 route to slot 1 and exit; only one
   Minecraft launches. Claimed the only fixes were `--unshare-net` (breaks internet)
   or dropping bwrap entirely.
3. **Recommendation:** Skip bwrap for Phase A, launch PolyMC directly, defer
   controller isolation to `feat/controlify-isolation`.

---

## Investigation — git archaeology

Rather than trust the "rip out bwrap" advice, we traced the actual history:

- `b24f91d` — original orchestrator rewrite: **no `--dev` bwrap at all**.
- `d5f060c` — **working era.** bwrap lived in `modules/instance_lifecycle.sh`
  (`_build_bwrap_command` / `spawn_instance`). It did `--dev /dev` **and then
  re-bound the GPU and friends back in:**

  ```
  bwrap --dev-bind / / --dev /dev \
    --dev-bind /dev/fuse /dev/fuse \
    --dev-bind /tmp/.X11-unix /tmp/.X11-unix \
    --dev-bind /tmp /tmp --dev-bind /home /home --dev-bind /run /run \
    --dev-bind /dev/dri /dev/dri \        # <-- GPU restored
    --dev-bind "$event_node" "$event_node" \
    --dev-bind "$js_node" "$js_node" \
    <--bind /dev/null masks for OTHER controllers> ...
  ```

  Controller isolation was done with `--bind /dev/null <other-pads>` — **NOT
  `--unshare-net`.** So internet worked, abstract sockets were shared, and the
  PolyMC SingleApplication primary launched all 4 JVMs normally. That forwarding
  is the *intended* multi-instance mechanism, not a bug.

- `420396f` — the windowing/position test stripped the launcher down to minimal.
- `38c4f99` — **Phase A** rebuilt `launchSlot` from scratch and **omitted the
  `/dev/dri` re-bind**, leaving bare `bwrap --dev-bind / / --dev /dev --proc /proc`.

## Conclusion — who was right

The **user's memory was correct.** bwrap genuinely worked before the windowing
test. It broke because the Phase A rewrite dropped the GPU re-bind, not because
bwrap is incompatible with this design.

- Handoff **Bug #1 is real** (GPU stripped) — but it's a regression, not a design flaw.
- Handoff **Bug #2 is a misdiagnosis** — SingleApplication forwarding was working
  fine in the d5f060c era with shared sockets; the slots only "exited" because the
  GPU bug killed the primary before the others could route to it.
- The "rip out bwrap" recommendation was therefore **wrong** and was not followed.

---

## The Fix

Restored the working-era dev re-binds (GPU + X11 + shm + fuse), guarded with
existence checks for portability, in **both** copies of `launchSlot`:

- `minecraftSplitscreen.sh` (prototype, ~line 171)
- `modules/launcher_script_generator.sh` (generator, ~line 338) — kept byte-identical
  in logic so the emitted launcher matches the prototype.

```bash
local -a bwrap_cmd=(bwrap --dev-bind / / --dev /dev --proc /proc)
# --dev /dev overlays an empty devtmpfs, wiping the device nodes that
# --dev-bind / / provided.  Re-bind the GPU (required by Qt xcb / LWJGL),
# X11 socket, shared memory, and FUSE back in, matching the known-working
# bwrap configuration from commit d5f060c (modules/instance_lifecycle.sh).
[[ -d /dev/dri ]]       && bwrap_cmd+=(--dev-bind /dev/dri /dev/dri)
[[ -e /dev/fuse ]]      && bwrap_cmd+=(--dev-bind /dev/fuse /dev/fuse)
[[ -d /dev/shm ]]       && bwrap_cmd+=(--dev-bind /dev/shm /dev/shm)
[[ -d /tmp/.X11-unix ]] && bwrap_cmd+=(--dev-bind /tmp/.X11-unix /tmp/.X11-unix)
```

The per-slot `XDG_RUNTIME_DIR=/tmp/polymc-runtime-slotN` hack from Phase A is
unnecessary (abstract sockets ignore it) but harmless, so it was left in place.

Both files pass `bash -n`. Committed as `d348bf1` and pushed to
`feat/gamescope-windowing`.

### New problem: all 4 instances silent
With windows working, the next gap surfaced: **no audio in any instance.**

**Root cause (confirmed by live Deck diagnostics, not assumed):**
- The Deck runs PipeWire with a PulseAudio compat shim. Audio sockets live at
  `/run/user/1000/pulse/native` (Pulse compat) and `/run/user/1000/pipewire-0`.
  `pactl info` → `Server String: /run/user/1000/pulse/native`.
- `launchSlot` overrides `XDG_RUNTIME_DIR` → `/tmp/polymc-runtime-slotN` (required to
  keep PolyMC SingleApplication sockets isolated so all 4 windows launch).
- PulseAudio/PipeWire clients discover the server via `$XDG_RUNTIME_DIR/pulse/native`.
  Repointing `XDG_RUNTIME_DIR` at the empty per-slot dir orphaned audio for every
  instance. `PULSE_SERVER` was unset → no fallback → silence.
- The socket file itself was already inside the sandbox (`--dev-bind / /` binds
  `/run`); only the env pointer was wrong.

### The fix — commit `1af8daf`
Added one env var to the bwrap block in **both** launchSlot copies (kept identical):

```bash
--setenv PULSE_SERVER "unix:/run/user/$(id -u)/pulse/native"
```

`$(id -u)` expands at launch as the running user (deck = uid 1000) — no hardcoded UID.
Both files pass `bash -n`. Pushed to `feat/gamescope-windowing`; Deck pulled to
`1af8daf` (PULSE_SERVER present in both files on the Deck).

**Expected behavior:** all 4 instances route to the same PipeWire/Pulse server and
mix together. `instance_creation.sh` already mutes *music* on instances 2–4 (instance
1 keeps it) to cut overlap; SFX play from all four (correct for splitscreen — each
player hears their own game's effects). Per-player device/channel routing is a
larger future change if desired, not needed for "sound works."
